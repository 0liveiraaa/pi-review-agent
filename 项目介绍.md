# Pi 复习助手 — 项目介绍

> 面向挑战杯评委和开发团队成员的完整项目说明。非技术背景读者建议先读本文了解项目全貌，再根据需要查阅 README（安装与使用手册）或 DESIGN.md（技术架构细节）。

---

## 1. 这是什么？解决什么问题？

一句话：**把你的课程笔记变成一个「私人 AI 助教」，它会出题、判题、讲解、追踪你的学习进度。**

传统复习的痛点是「被动重读笔记」——翻一遍课本、划几条线，大脑其实没有真正参与。认知科学里有一个基本结论：**主动回忆（active recall）远比被动重读有效。** 但一个人看书时很难持续给自己出题、即时判对错、根据薄弱点调整节奏。

Pi 复习助手做的事情就是把这个过程自动化。你给它一份笔记（Markdown 格式），它生成一份结构化的「复习资料包」。进入复习后，AI 根据资料包出题，你在终端界面里答题，AI 即时判题并给出解析。每次答题后，系统更新你的错题本、知识点掌握度和学习画像。下次打开，你知道自己哪些地方薄弱、今天该优先复习什么。

**它不是聊天机器人。** 聊天机器人给你一个对话框，你自由输入，它随意输出——但这种自由对复习场景是危险的。Pi 复习助手的交互通过固定的按钮和选项（而非自由文本）来完成：出题、判题、归档、选择下一步——每一个环节都有严格的格式校验，AI 不能「临场发挥」出界。

**它不是网课平台。** 网课平台提供统一内容（视频、讲义），你是被动接收者。Pi 复习助手不生产课程内容——它把你自己的笔记（或同学分享的资料包）变成可复习、可追踪的练习系统。内容属于你，学习数据也属于你。

## 2. 核心功能

### 2.1 从笔记到「复习资料包」(Profile)

一份 Markdown 笔记丢进去，系统将其转化为一套结构化的复习资料包。这个转化过程有 8 个步骤，每个步骤产出特定的文件：

| 步骤          | 产出物                   | 说明                                                      |
| ------------- | ------------------------ | --------------------------------------------------------- |
| 1. 通读源资料 | —                       | AI 逐个阅读源文件，判断科目类型、章节划分、知识点密度     |
| 2. 科目描述   | `subject.md`           | 科目概述、考试目标、出题风格偏好                          |
| 3. 章节切分   | `chapters/*.md`        | 按章/节整理的规范化材料，每个小节聚焦 1-3 个关联知识点    |
| 4. 知识点索引 | `knowledge_index.json` | 每个知识点的 ID、名称、别名、难度基线、常见误区、出题提示 |
| 5. 概念卡片   | `cards/*.md`           | 每个知识点的精炼卡片：定义、关键要点、代码示例、常见误区  |
| 6. 考点总结   | `exam_points/*.md`     | 每章的考点清单、重点难点标注、典型例题、复习建议          |
| 7. 源文件映射 | `source_map.json`      | 每个产出文件到原始笔记的溯源映射                          |
| 8. 质量报告   | `quality_report.md`    | 整体评估、严重问题、待改进项、低置信度标注、启用建议      |

整个过程由 `/review-init` 命令触发。生成完毕后，AI 不会自动启用资料包——你必须先查看质量报告，确认无误后手动确认启用。如果质量不达标，通过 `/review-fix` 修订，或者补充源资料后重新生成。

**以 C++ 面向对象程序设计课程为例**：一份覆盖 20 章、74 个知识点的笔记（从 C++ 编译过程、变量类型、指针/引用到构造函数、运算符重载、虚函数多态、动态内存管理），经过 8 步管线生成包含卡片、章节材料、考点总结的完整资料包。每个知识点标注了难度基线（如 `copy_constructor` 为 M-U、`virtual_function` 为 M-A）和具体出题提示（如「考拷贝构造与赋值的区别，陷阱：浅拷贝导致的 double-free」）。

### 2.2 三种复习模式

系统提供三种模式，对应三个不同的复习场景：

| 模式                    | 场景                         | 流程                                                             |
| ----------------------- | ---------------------------- | ---------------------------------------------------------------- |
| **概念卡片+练习** | 背概念、记定义、考前巩固基础 | 先看知识点卡片（核心定义、关键要点、代码示例） → 做相关题目巩固 |
| **直接练习**      | 刷题冲刺、检验掌握度         | 先看章节考点总结 → 直接出题作答                                 |
| **章节笔记学习**  | 系统学习、查漏补缺           | 先阅读章节材料（逐小节） → 出理解性问题                         |

三种模式共享同一套后续流程：出题 → 作答 → 判题 → 归档 → 选择下一步。

**C++ 场景举例**：

- 卡片模式：看「拷贝构造函数」卡片（定义、浅拷贝 vs 深拷贝、代码示例），确认理解了再做题。
- 练习模式：直接做「写出一个类的拷贝构造函数和赋值运算符重载」的选择/简答题。
- 章节模式：通读第 9 章「对象的拷贝」整章材料，从拷贝构造到赋值函数逐步出题。

### 2.3 五级难度体系

题目难度 = **知识广度 × 认知层次**。广度指涉及几个知识点（单一 / 多概念 / 知识链），认知层次指对知识的加工深度（回忆 / 理解 / 分析）。

| 级别 | 缩写 | 广度 × 认知   | 含义                | C++ 示例                                                                               |
| ---- | ---- | -------------- | ------------------- | -------------------------------------------------------------------------------------- |
| 1 级 | S-R  | 单一 × 回忆   | 单个概念，记忆/识别 | 「什么是构造函数？它的调用时机是什么？」                                               |
| 2 级 | S-U  | 单一 × 理解   | 单个概念，理解/辨析 | 「构造函数可以设置默认参数吗？如果可以，语法如何？如果不能，为什么？」                 |
| 3 级 | M-U  | 多概念 × 理解 | 2-3 个关联概念比较  | 「构造函数和析构函数在调用时机和顺序上有什么不同？哪些场景下析构必须为虚？」           |
| 4 级 | M-A  | 多概念 × 分析 | 多概念综合推理      | 「分析以下代码中构造和析构的顺序，指出其中潜在的内存错误。」                           |
| 5 级 | C-A  | 知识链 × 分析 | 知识链综合运用      | 「设计一个支持多态的类层次结构，使用虚析构函数和智能指针管理资源，说明你的设计决策。」 |

难度可以手动选择（「我今天只想练 S-R」），也可以选「自动」——系统根据知识点的难度基线和你的历史正确率动态调整。

### 2.4 题型

| 题型     | 适用难度      | C++ 举例                                                                                                  |
| -------- | ------------- | --------------------------------------------------------------------------------------------------------- |
| 正误判断 | S-R, S-U      | 「拷贝构造函数默认执行深拷贝。对/错？」                                                                   |
| 单项选择 | S-U, M-U, M-A | 「以下哪个选项会导致浅拷贝问题？A. 默认拷贝构造 B. 自定义拷贝构造（深拷贝） C. 移动构造 D. 禁用拷贝构造」 |
| 多项选择 | M-U, M-A      | 「以下哪些行为会触发析构函数调用？（多选）」                                                              |
| 简述题   | M-A, C-A      | 「简述 RAII 的三个核心原则，并给出一个 C++ 代码示例。」                                                   |

除了通用题型，系统支持按科目适配。编程课使用代码阅读和输出/结果判断题，数学课使用推导/证明题，文科课使用论述题。不同科目有不同的出题模板和判题标准（rubric）。

### 2.5 多层复盘

每道题做完不是终点。系统提供三层复盘，逐步深入：

| 层次 | 名称     | 内容                                                | 触发方式                  | 类比           |
| ---- | -------- | --------------------------------------------------- | ------------------------- | -------------- |
| L1   | 即时解析 | 正确答案、你的答案对比、关键知识点讲解              | 每题自动给出              | 对答案         |
| L2   | 深度追问 | 针对本题的拓展讨论、易错点分析、相关概念联系        | 题后菜单选「💬 追问」     | 追问为什么     |
| L3   | 全景梳理 | 本题在知识链中的位置、前置/后继知识点、整体掌握脉络 | 每 5-10 题选「📝 做总结」 | 串起整条知识线 |

此外，退出复习时生成一份完整的**会话总结报告**（时间、正确率、错题清单、薄弱环节、下一步建议）。报告保存在你的私有数据目录中，可随时回顾。

### 2.6 智能追踪（三条长期数据线）

系统在你每次复习时自动维护三条数据线，跨会话持久化：

**错题本（`wrong_book.json`）**：

- 答错的题目自动进入错题本
- 后续复习时错题本中的知识点优先出现
- 同一知识点连续答对多次后自动降低优先级
- 跨所有资料包共享（不管复习哪门课，你的错题都在）

**知识链路（`knowledge_chains.json`）**：

- 知识点之间的前置/后继关系（如「拷贝构造函数」的前置是「浅拷贝 vs 深拷贝」，后继是「移动语义」）
- 每个知识点的掌握度随时间衰减：久未复习的知识点会自动降低掌握度
- 掌握度低于阈值的知识点标记为薄弱

**学习画像（`learning_profile.json`）**：

- 每个科目的长期画像：正确率变化趋势、薄弱知识点排行榜、近期会话总结摘要
- 每次会话结束后由总结报告自动提炼更新
- 下次打开 `/review` 选择该科目时展示画像摘要

## 3. 实现方式

### 3.1 代码与 AI 的职责边界

系统由两部分协作运行：**代码**和 **AI（大语言模型 Agent）**。它们有严格而清晰的职责划分。

**代码负责的（稳定、可重复、不出错）**：

- 资料包的目录结构、路径安全校验（禁止写入非 draft 资料包）
- 卡片、章节材料、考点总结、题目的 TUI（终端界面）渲染
- 答题记录归档（写到正确的目录和文件）
- 状态更新（错题本、知识链、学习画像）
- 所有工具的输入参数格式校验（schema）
- 交互流程控制（各节点的出入条件和路由）

**AI 负责的（需要理解、需要灵活、需要内容生成）**：

- 理解你的笔记内容，抽取出知识结构
- 根据资料包出题（选择合适题型、难度、构造题干和选项、设计陷阱）
- 判题（判断你的答案是否正确、指出错误原因）
- 讲解（L1 解析、L2 追问、L3 全景梳理）
- 根据你的反馈修订资料包内容

这个边界是全项目最核心的设计决策。可以概括为：**凡是每次用户体验必须一致的部分，由代码保证；凡是需要理解语义做出判断的部分，交给 AI。**

### 3.2 技术栈

项目以 **pi-agent 扩展（extension）** 形态运行，不启动独立进程或服务器。所有代码和资源文件驻留在 `workspace/` 目录下，由 pi-agent 加载。

**运行环境**

| 组件     | 版本/来源                                  | 说明                                                                                                                                                                                                                 |
| -------- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Node.js  | ≥22                                       | JavaScript/TypeScript 运行时。团队全部代码在此之上运行                                                                                                                                                               |
| pi-agent | `@earendil-works/pi-coding-agent` v0.78+ | 命令行 AI 助手平台。提供扩展注册、Agent 上下文管理、MCP 工具接入等基础能力                                                                                                                                           |
| pi-tui   | `@earendil-works/pi-tui`                 | 终端 UI 框架（Text User Interface）。提供 SelectList（选择列表）、Editor（文本输入）、Container（布局容器）、DynamicBorder（动态边框）等组件。**所有用户交互界面均由这些组件在终端内渲染，不依赖浏览器或 GUI** |

**核心依赖**

| 依赖    | 用途                     | 对团队的意义                                                                                                                                  |
| ------- | ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| typebox | JSON Schema 运行时校验   | 每个工具（见 3.3 节）的输入参数在调用前由 typebox 验证：类型不匹配、缺少必填字段、值超出范围——直接拒绝执行。这是「AI 不能乱来」的代码层保障 |
| marked  | Markdown → TUI 文本转换 | 资料包中的 Markdown 文件（卡片、章节、考点）转换为终端可显示的纯文本。不涉及 HTML 渲染，只做 AST 解析和纯文本输出                             |

**代码组成**

| 层级       | 位置                                     | 语言             | 规模          | 职责                                                                                                     |
| ---------- | ---------------------------------------- | ---------------- | ------------- | -------------------------------------------------------------------------------------------------------- |
| 扩展入口   | `workspace/extensions/review/index.ts` | TypeScript       | ~1000 行      | 注册 3 个命令 + 9 个工具 + session hook。所有用户交互的 TUI 组件逻辑（滚动、分页、输入校验）都在此文件中 |
| 核心库     | `workspace/lib/*.mjs`                  | JavaScript (ESM) | 8 个模块      | 业务逻辑：资料包管理、题目处理、状态持久化、引擎调度、卡片匹配、章节扫描                                 |
| Skill 规则 | `workspace/skills/*/SKILL.md`          | Markdown         | 14 个 skill   | AI 行为规范文档。review-core 是主规则（强制注入每次 Agent 调用），其余按任务类型按需加载                 |
| 资料包模板 | `workspace/docs/review-kit/skills/`    | Markdown         | 与 Skill 镜像 | 面向资料包维护者的使用说明和模板，与运行时 Skill 保持同步                                                |

**核心 Skill 体系**

Skill 是一份 Markdown 文档，定义了 Agent 在特定场景下的行为规范、工具调用顺序、输出格式和硬性约束。它不是给用户读的——它在运行时被注入到 Agent 的上下文中，Agent 必须遵守。

| Skill                              | 角色       | 关键约束                                                                                     |
| ---------------------------------- | ---------- | -------------------------------------------------------------------------------------------- |
| `review-core`                    | 主规则     | 运行时契约、工具路由、模式流程、资料包生命周期。**每次命令都强制注入，Agent 不能跳过** |
| `review-question`                | 出题       | 五级难度定义、题型模板、题目 JSON schema、难度自适应策略                                     |
| `review-grade`                   | 判题       | L1 解析格式、对错判断标准、按题型差异化评分                                                  |
| `review-discuss`                 | 追问       | L2 讨论规范、C++ 代码示例风格、深度和广度边界                                                |
| `review-summary`                 | 复盘       | 每题复盘 JSON 格式、会话总结报告模板、学习画像更新字段                                       |
| `review-init`                    | 初始化     | 8 步生成流程、各步骤产出物规范、低置信度标注规则                                             |
| `review-fix`                     | 修订       | 资料包修订流程、draft 安全写入、质量报告重写规范                                             |
| `review-profile-structure`       | 资料包结构 | 目录布局、文件命名规则、profile.json schema                                                  |
| `review-profile-index`           | 知识点索引 | knowledge_index.json 结构、字段填写规范、粒度测试                                            |
| `review-profile-cards`           | 卡片       | 卡片 frontmatter 字段、内容模板、批量生成策略                                                |
| `review-profile-exam-points`     | 考点       | 考点总结结构、重点难点标注、例题选择原则                                                     |
| `review-profile-quality`         | 质量审核   | 质量六维评分、严重问题识别、启用建议生成                                                     |
| `review-profile-training-assets` | 训练资产   | problem_templates 结构、题型模板、rubric 编写                                                |

**Agent 交互范式：ReAct**

Agent 与系统的交互遵循 ReAct（Reasoning + Acting）范式：

```
Agent 观察（Observe）→ 思考（Think）→ 调用工具（Act）→ 接收工具返回 → 观察 → 思考 → ...
```

每一步「思考→行动→观察」为一个 ReAct 循环。Agent 在节点内部的 react 循环（如反复要提示/追问）就是典型的 ReAct 模式。离开节点的条件由代码校验通过后触发——不满足条件则 Agent 始终留在当前节点内继续 ReAct。

### 3.3 三条命令线 + 九大工具

系统对外暴露三个命令，内部提供九个工具（工具是 AI 调用代码的唯一方式）。三条命令是三条独立的节点图，每条线有各自的节点顺序和出入条件。

**九大工具一览**：

| 工具                      | 用途                       | 所属命令线                        | 关键硬约束                                                            |
| ------------------------- | -------------------------- | --------------------------------- | --------------------------------------------------------------------- |
| `review_card`           | 渲染知识点卡片             | `/review`                       | 卡片不存在时返回`card_found: false`，不阻塞                         |
| `review_exam_points`    | 渲染章节考点总结           | `/review`                       | 考点文件缺失时返回空内容                                              |
| `review_chapter`        | 渲染章节或小节材料         | `/review`                       | 支持逐小节选择                                                        |
| `review_answer`         | 渲染题目并收集用户答案     | `/review`                       | 题型对应的答案格式由代码校验（选择必须从选项范围取）                  |
| `review_archive`        | 归档答题记录，更新全量状态 | `/review`                       | `is_correct` 必须为 boolean，`knowledge_points` 必填              |
| `review_turn_action`    | 每道题后的操作菜单         | `/review`                       | **每道题必经**，用户必须选择一个操作，不可跳过                  |
| `review_summary`        | 保存会话总结，更新学习画像 | `/review`                       | 写入`_user/summaries/`，同时更新 `_user/learning_profile.json`    |
| `review_profile_write`  | 写入文件到 draft 资料包    | `/review-init`, `/review-fix` | **拒绝非 draft 资料包**，路径穿越（`..`）被封堵               |
| `review_profile_enable` | 启用 draft 资料包          | `/review-init`, `/review-fix` | **只接受 draft→active**，若为 revision draft 则先归档旧 active |

每个工具的输入参数都有严格的格式定义（通过 typebox 做 JSON Schema 运行时校验）。AI 调用工具时必须填入所有必填字段且类型匹配，否则工具拒绝执行。

#### `/review` 线（复习主循环）

4 个节点 + 1 个强制路由点，用户判断何时退出。

```
用户选择（profile → 模式 → 范围 → 题型 → 难度）
```

### 3.4 资料包生命周期

```
draft（草稿/可编辑）
   │
   │ review_profile_enable（用户确认启用）
   ▼
active（已启用/可复习/不可编辑）
   │
   │ 被修订版替代时
   ▼
archived（已归档/保留在磁盘/可回滚）
```

- **draft**：`/review` 不显示，`/review-fix` 可编辑。所有资料包从 draft 起步。
- **active**：`/review` 可选择进入复习。**不可直接编辑**——如需修改，通过 `/review-fix` 创建 revision draft。
- **archived**：旧版本保留在磁盘，不参与 `/review` 和 `/review-fix` 的列表。如果新修订版有问题，可以从 archived 手动回滚。

**修订命名规则**：修订 active 资料包时，新 draft 名称回到原始 root subjectId 重新编号——不基于当前 active 叠加前缀。例如 `cpp-oop__draft_20260621`、`cpp-oop__draft_20260622_v2`。`profile.json` 中记录 `revisionOf`（被修订的版本）、`revisionRoot`（原始 subjectId）、`revisionNumber`（当天版本号）。

### 3.5 数据落点

用户数据分为两类：**可分享的（资料包本身）** 和 **私有的（个人学习数据）**。

#### 资料包数据（`review_profiles/{subjectId}/`）

```
review_profiles/{subjectId}/
├── profile.json              ← 元数据（subjectId / name / status / revision 链）
├── subject.md                ← 科目描述
├── knowledge_index.json      ← 知识点索引
├── cards/                    ← 概念卡片（*.md）
├── chapters/                 ← 章节材料（*.md）
├── exam_points/              ← 考点总结（*.md）
├── source_map.json           ← 源文件映射
├── quality_report.md         ← 质量报告
│
└── _user/                    ← 私有（不随资料包分享/导出）
    ├── summaries/            ← 会话总结报告
    │   └── {session_id}_总结.md
    └── learning_profile.json ← 该科目的学习画像
```

**`_user/` 目录是关键边界**：它位于资料包目录内（方便按科目关联），但内容属于用户私有数据。当你导出资料包分享给同学时，`_user/` 内容不会被打包。同学导入后有自己的 `_user/` 目录。

#### 全局状态数据（`state/` 和 `archive/`）

与 `_user/` 共存，当前两套路径并行写入：

| 路径                                         | 内容                                  | 数据归属   | 迁移方向                                                |
| -------------------------------------------- | ------------------------------------- | ---------- | ------------------------------------------------------- |
| `archive/sessions/{sid}/`                  | 每次答题的详细记录（JSON + Markdown） | 全局共享   | 当前仍写全局，后续考虑按 profile 分离                   |
| `archive/summaries/`                       | 历史会话总结                          | 全局共享   | 已在`_user/summaries/` 新增并行写入                   |
| `state/wrong_book.json`                    | 错题本                                | 全局共享   | 跨科目共用，保持全局                                    |
| `state/knowledge_chains.json`              | 知识链状态                            | 全局共享   | 跨科目共用，保持全局                                    |
| `state/learning_profiles/{subjectId}.json` | 学习画像（旧路径）                    | 全局共享   | 已迁移到`_user/learning_profile.json`，旧路径兼容回退 |
| `_user/summaries/`                         | 会话总结（新路径）                    | 按科目私有 | **优先级高于全局 summaries**                      |
| `_user/learning_profile.json`              | 学习画像（新路径）                    | 按科目私有 | **优先级高于旧 learning_profiles**                |

规则：

- `review_summary` 写入 `_user/summaries/`，同时更新 `_user/learning_profile.json`
- `/review` 启动后优先读取 `_user/learning_profile.json`
- 若私有画像不存在，回退到旧的 `state/learning_profiles/{subjectId}.json`
- 两套并行路径最终统一到 `_user/`，旧路径保留兼容但不作为主要数据源

## 4. 设计思路

### 4.1 「代码稳定，Agent 灵活」——全项目核心哲学

Agent（大语言模型驱动的智能体）是强大的——它理解语义、生成内容、做出判断。但它也是不稳定的——同一个问题问两次，可能得到不同的回答。对于复习场景，这种不稳定是致命的：如果一道题的判题标准前后不一致，用户就无法建立信任。

解决方式不是「让 Agent 更稳定」（这是通用 AI 问题，不是我们可控的），而是**把稳定性的责任从 Agent 身上拿掉，交给代码**。

- 代码来定义知识的组织结构（`knowledge_index.json` 的 schema 不允许 shape 变形）
- 代码来渲染界面（卡片、章节、考点、题目、操作菜单的 TUI 布局是固定的）
- 代码来写入数据（归档、状态更新、路径校验——不是 Agent 写文件，是 Agent 调用代码工具，代码来写）
- 代码来做格式校验（typebox 运行时验证 Agent 传给工具的 JSON 参数）

Agent 只做它擅长的事：从资料中理解知识结构、生成有区分度的题目、判断用户答案的对错、针对错因给出解释。这些任务天然需要灵活性和语义理解，没有固定答案模板可套。

切好了这道边界，就同时得到了两样东西：交互的可靠性（代码保证）和内容的灵活性（Agent 保证）。两者不冲突。

### 4.2 资料包是核心产品资产，不是临时缓存

资料包不是「AI 处理笔记时产生的临时文件」。它是一个独立、自包含、可迁移的知识产品。理由：

- **可迁移**：把 `review_profiles/{subjectId}/` 目录整体移到另一台机器，所有资料和结构保持不变。
- **可分享**：一个同学生成的 C++ OOP 资料包，其他同学导入后可以直接使用。不需要各自重新生成（省 token、省时间）。
- **可版本化**：资料包遵循 `draft → active → archived` 生命周期，修订历史完整可追溯。旧版本不会丢失，可以回滚。
- **可审查**：人可以直接打开 `quality_report.md` 评估质量，不需要理解任何代码。

推广这个产品的能力，很大程度上取决于用户能否获取、制作和分享高质量的资料包。一个用户花 token 和时间生成的精品资料包，不应该停留在「自己的机器上的一条临时记录」——它应该可以被传播。

### 4.3 结构化 JSON + Schema 校验：不让 AI 自由发挥

在「AI 生成题目」这个场景里，一种常见的做法是：AI 生成自然语言文本，代码从文本中用正则表达式提取题型、选项、答案。这种做法的问题是脆——AI 多换一个行、多加一个空格、换一种表述，正则就失效。

Pi 复习助手的做法是：**所有 AI 与代码之间的数据交换都是结构化 JSON，且入口有 schema 校验。**

```
AI 生成题目 → 输出 JSON：
{
  "type": "choice",
  "question_text": "...",
  "options": ["A. ...", "B. ...", "C. ...", "D. ..."],
  "correct_answer": "B",
  "knowledge_points": ["copy_constructor"],
  "difficulty": "M-U",
  "explanation_l1": "...",
  "source_basis": "chapters/9.1-对象的拷贝.md"
}
    ↓
review_answer 工具接收此 JSON
    ↓
typebox 校验：type 是 "choice" → 选项数组至少 2 个 → correct_answer 在选项范围内 → 通过
    ↓
TUI 渲染：题目文字 + 四个选项 + 用户选择
```

如果 AI 输出的 JSON 缺了 `correct_answer` 字段，或 `type` 填了一个不存在的值，typebox 拒绝执行，AI 必须修正后重试。这道硬约束让「AI 不稳定」变成了「AI 在可控范围内试错」。

### 4.4 质量前置，诚实标注

AI 从笔记生成资料包时，有些内容它可以确信（比如从明确的小标题提取章节结构），有些内容则不确定（比如两个概念的关系它不确定是并列还是前后置）。如果不对这些不确定做标记，用户在不知情的情况下启用有缺陷的资料包，后续的复习体验会被放大。

因此：

- **不确定的内容必须标注**：chapters 文件末尾加 `<!-- UNCERTAIN_STRUCTURE: 原因 -->`，source_map 里标注 `uncertain_mappings`
- **质量报告阻断「带病启用」**：`quality_report.md` 里列出严重问题、待改进项、低置信度项，并给出启用建议（「可以启用」「修复严重问题后启用」「补充资料后重新生成」）
- **用户必须主动确认启用**：资料包生成完毕后保持 `draft` 状态，用户必须查看质量报告后手动确认启用。代码不替用户做这个判断

一句话：**不让坏资料放大成坏体验。**

### 4.5 Demo 是产品入口，不是样例代码

内置的学习方法 Demo profile 不是为了展示「代码怎么写」——它是为了让用户在 1 分钟内理解整个产品的交互闭环：

> 选择 profile → 看到卡片 → 做一道题 → 看到判题解析 → 选择下一步 → 做总结 → 看到学习画像

走完这一个循环，用户就理解了「卡片、考点、章节、答题、总结」五者的关系。Demo 内容刻意选择跨学科中性主题（学习方法和认知科学），避免 C++ 背景成为理解门槛。

## 5. 后续演进

### 5.1 一个正在成型的方法论：Loop 与 Gragh

#### 想法

当前 AI Agent 领域的主流范式是：Agent 读取一份 Skill 文档（规则和流程说明），然后在一个开放上下文中自主执行任务——读文件、调工具、生成内容。这个范式的核心问题是：**Agent 在执行过程中缺乏代码层的硬约束**——它能调哪些工具、什么时候必须离开当前阶段、什么条件下才能进入下一步，全靠自然语言在 Skill 文档里「建议」。

Loop 工程是一种更进一步的封装范式。核心思路是：**把开放式 Agent 执行空间拆分为有限个特化闭环（Loop），每个闭环有严格的出入条件和专属工具集，闭环之间用有向图组织。**

一个 Loop（节点）的要素：

| 要素             | 含义                            | 由谁负责           |
| ---------------- | ------------------------------- | ------------------ |
| 目标（效用函数） | 本节点要达成什么                | Skill 定义         |
| 上下文           | 本节点可读取的信息范围          | 代码注入           |
| 可调用工具       | 本节点内 Agent 可以使用哪些工具 | 代码注册           |
| 进入条件         | 满足什么条件才能进入本节点      | **代码校验** |
| 离开条件         | 满足什么条件才能离开本节点      | **代码校验** |
| 内部 Hook        | 进入/离开时触发的代码逻辑       | 代码执行           |

Agent 在节点内部仍采用 ReAct 范式（观察→思考→行动→观察→…）循环，直到满足离开条件才被代码释放到下一节点。不满足条件则永远留在当前节点内循环。

多个节点以有向图连接：

```
[节点 A] ──→ [节点 B] ──→ [节点 C]
   │                          │
   └────────→ [节点 D] ←──────┘
```

有向图支持分支和汇合，不同的起始节点可以定义不同的 Agent 工作线，最终实现多 Agent 系统（每条线由不同的 Agent 实例驱动）。

更重要的是，节点支持封装：**多个节点可以被封装为一个更高层级的节点**。对外暴露统一的进入/离开条件，内部则是有向子图。这种封装可以不断递归，就像面向对象中对类的组合——你不需要关心一个类的内部实现，只需要知道它的接口（进入条件、离开条件、可调用的外部工具）。

#### 现在怎么体现的

当前 Pi 复习助手的代码中虽然还没有显式的 Loop 引擎，但三条命令线的结构已经暗合了这个范式：

- **每条线是一张有向图**（见第 3.2 节的节点图）
- **每个节点有专属工具集**（前置展示节点的工具 ≠ 出题节点的工具 ≠ 归档节点的工具）
- **有硬性出入条件**（`review_profile_write` 拒绝非 draft 写入、`review_archive` 要求 `is_correct` 为 boolean、`review_turn_action` 每题必经不可跳过）
- **用户判断是主要离开条件**（用户觉得练够了 → exit；用户觉得资料质量 OK → enable；代码不做内容质量判断）
- **节点内部支持 react 循环**（出题节点里用户反复要提示/追问，Agent 在本节点内持续响应）

这些设计选择不是偶然的——它们来自反复踩坑后的经验：「让 Agent 自由规划执行路径」会导致跳过关键步骤、工具顺序错乱、上下文溢出。逐步约束后，才收敛到现在的节点结构。

#### 之后怎么做

Loop 工程从隐式规范走向显式引擎，分三步：

**第一步：抽象 Loop 节点类型**。在代码层提供节点定义接口（目标函数、上下文注入器、工具注册表、进入条件校验器、离开条件校验器、Hook 注册），从当前硬编码的 switch-case 升级为可配置的节点注册。

**第二步：有向图路由引擎**。节点的连接关系不再散落在 Agent 的 Skill 文档里，而是由代码层的有向图定义。图引擎负责：入度/出度检查、条件不满时自动循环、记录节点间跳转日志。

**第三步：节点封装与组合**。把 `/review` 线的 4 节点子图封装为一个「单题复习」节点，把 `/review-init` 线的 8 步骤封装为一个「资料包生成」节点。这两个封装节点可以作为更复杂工作流（如「生成资料包 → 自动质量检查 → 抽样出题验证 → 启用」）的子节点。

这个方向不改变现有的产品功能和用户交互——它改变的是内部的扩展性和可维护性。当需要新增第四条命令线（如真题训练线）时，不再需要写一个全新的 agent prompt 和状态机，而是定义一张新的节点图、注册每个节点的出入条件和工具即可。

### 5.2 近期路线（当前优先）

- **安装和首次体验做稳**：安装脚本自带诊断（doctor），检测 Node 版本、pi 版本、扩展入口、资料包数量、数据目录位置。常见错误（残留 `workspace/.pi`、Node 版本过低、资料包 schema 不合法）给出可执行的修复提示。
- **资料包生成质量强化**：`/review-init` 的质量报告拆分为结构完整性、知识点粒度、卡片覆盖率、章节覆盖率、考点覆盖率、source map 完整性六个维度。支持按质量报告条目逐项修订（不只是自由文本反馈）。
- **资料包导入导出**：定义资料包包格式（如 `.review-profile.zip`），支持 `/review-import` 和 `/review-export`。导出的资料包不包含 `_user/` 私有数据。同学之间分享资料包，不用各自重新生成（省 token）。

### 5.3 中期路线

- **「今日推荐」复习队列**：打开 `/review` 后不需要手动选择章节——系统根据学习画像、错题本、卡片进度、知识链状态自动生成推荐复习队列。排序因素：未看过的卡片 → 最近答错的知识点 → 低掌握度 → 久未复习 → 真题高频考点。
- **遗忘曲线策略**：知识点的掌握度随时间衰减（可配置衰减曲线），系统根据上次复习时间和当前掌握度决定复习优先级。
- **学科适配层**：资料包增加 `subject_type` 字段（programming / math / language / humanities），不同学科有不同题型模板和判题标准（rubric）。编程课支持代码阅读和 bug 定位，数学课支持推导和证明，文科课支持论述和材料分析。

### 5.4 长期愿景：从「复习插件」到「个人知识学习运行时」

当前的 Pi 复习助手定位为课程复习插件。但如果把视野拉长，它真正有潜力的部分不是「出题」——而是**把任意知识材料转化为可学习、可追踪、可复盘的结构化资料包，并用同一套交互运行时驱动学习过程**。

这个抽象可以覆盖更广泛的知识产品类型：

| 资料包类型       | 知识载体                                      | 核心交互                                             |
| ---------------- | --------------------------------------------- | ---------------------------------------------------- |
| 课程复习（当前） | 章节笔记 → 知识点索引 → 卡片/考点           | 卡片练习、直接练习、章节学习                         |
| 论文阅读         | paper 元数据 → 问题/方法/证据/局限 → claims | 速读摘要、精读章节、方法拆解、证据链检查、批判性问答 |
| 真题训练         | 历年真题 → 知识点映射 → 风格模板            | 随机抽题、按章节/题型训练、真题风格仿真出题          |
| 读书笔记         | 书目大纲 → 章节笔记 → 核心论点              | 顺序阅读、观点提炼、概念卡片、读后复述               |
| 技术文档         | 概念索引 → API 参考 → 工作流示例            | 概念学习、API 查询、示例理解、实操任务               |
| 语言学习         | 词汇表 → 语法点 → 阅读材料                  | 单词卡片、语法辨析、翻译练习、阅读理解               |

核心思路是「换不同的资料包，就像换不同的课本」——底层交互运行时、状态追踪、归档复盘是一套，资料包的结构和交互模式按类型适配。当这套框架覆盖了课程复习之外的第一个非课程类型（如论文阅读），项目就从「复习插件」升级成了「个人知识学习运行时」。

### 5.5 暂缓事项（诚实说明）

以下能力有价值，但不建议近期优先投入：

| 事项                      | 暂缓原因                                                                                                                                    |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| PDF / Word / PPT 原生导入 | 先把 Markdown/txt 的资料包生成质量做高。非 Markdown 格式可通过外部工具转换后导入。过早做多格式解析会分散核心管线精力                        |
| Web UI                    | 当前产品依托 pi-agent TUI（终端界面）。过早做 Web 会引入前后端分离、状态同步、认证等独立问题，偏离「终端学习工具」的定位                    |
| 云同步                    | 先把本地数据目录和导入导出做稳。本地文件天然支持 Git 管理、U 盘拷贝、网盘同步                                                               |
| 多人班级看板              | 除非明确要做教师管理端，否则会把产品带向 LMS（学习管理系统）方向，与「个人工具」定位冲突。资料包分享 + 个人数据本地方案已覆盖同学间使用场景 |
