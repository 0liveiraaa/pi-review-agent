# 19.2 虚机制的作用

## 核心概念

- **虚机制**：C++ 实现动态多态的语言机制，通过虚函数和虚函数表让派生类重写基类接口。
- **子类型化**：派生类对象被视为基类类型的子类型，可以在基类指针/引用处使用。
- **面向接口编程**：客户代码依赖稳定接口，具体行为由对象动态类型决定。

## 知识点详解

### 虚机制只是实现方式

虚机制（虚函数表、虚函数调用）是实现动态多态的技术手段。真正重要的问题是：**动态多态在程序设计中有什么价值**。

核心结论：虚机制的价值不在于"调用更复杂"，而在于**通过子类型化适应变化**。

### 保持客户代码不变

设有基类 `A` 声明虚函数 `func()`，派生类 `B`、`C` 分别重写：

```cpp
class A {
public:
    virtual ~A() = default;
    virtual void func();
};

class B : public A { void func() override; };
class C : public A { void func() override; };
```

客户代码：

```cpp
void client(A* p) {
    p->func();
}
```

`client` 不需要知道 `p` 真实指向 `B`、`C`，还是未来新增的 `D`。只要新类继承 `A` 并重写 `func()`，客户代码就能在不修改的情况下使用新实现。

这就是虚机制的核心价值：**保持访问接口不变，让实现可以变化**。

### 通过子类型化组织变化

虚机制的本质是**通过子类型化组织变化**：

| 组件 | 角色 |
|------|------|
| `A::func()` | 定义稳定接口 |
| `B::func()` | 变化实现 1 |
| `C::func()` | 变化实现 2 |
| `D::func()` | 未来新增变化实现 |

新增一种变化时，通常只需要新增一个子类，而**不必修改大量客户代码**。这是 OCP（开闭原则）的一种体现。

### "看一行代码可能不知道结果"

```cpp
p->func();
```

如果 `func` 是虚函数且 `p` 是基类指针，那么只看这一行代码无法知道最终执行哪个函数体。原因：`p` 的动态类型可能是任何派生类，包括未来才会写出的派生类。

这种"不知道"不是缺点，而是**有意的设计弹性**：客户代码面向稳定接口，具体行为由对象动态类型决定。

### 面向接口编程

```cpp
void drawAll(vector<Shape*>& shapes) {
    for (Shape* s : shapes) {
        s->draw();
    }
}
```

这段代码只依赖 `Shape::draw()` 接口，不依赖 `Circle`、`Rectangle`、`Triangle` 的具体实现。未来增加 `Polygon`，只要它也是 `Shape` 的子类型，`drawAll` 就不需要改。

## 关键示例

### 客户代码不修改即可扩展

```cpp
#include <iostream>
#include <vector>
using namespace std;

class Animal {
public:
    virtual ~Animal() = default;
    virtual void speak() const { cout << "..." << endl; }
};

class Dog : public Animal {
public:
    void speak() const override { cout << "Woof!" << endl; }
};

class Cat : public Animal {
public:
    void speak() const override { cout << "Meow!" << endl; }
};

// 客户代码——面向 Animal 接口
void greeting(const Animal& a) {
    a.speak();
}

int main() {
    Dog d;
    Cat c;
    greeting(d);  // Woof!
    greeting(c);  // Meow!
    
    // 新增 Bird 只需新增子类，greeting 不需要改
    return 0;
}
```

### Shape 绘图示例

```cpp
#include <iostream>
#include <vector>
using namespace std;

class Shape {
public:
    virtual ~Shape() = default;
    virtual void draw() const = 0;  // 纯虚函数
};

class Circle : public Shape {
public:
    void draw() const override { cout << "draw Circle" << endl; }
};

class Rectangle : public Shape {
public:
    void draw() const override { cout << "draw Rectangle" << endl; }
};

// 面向接口
void drawAll(const vector<Shape*>& shapes) {
    for (auto s : shapes) s->draw();
}

int main() {
    Circle c;
    Rectangle r;
    vector<Shape*> shapes{&c, &r};
    drawAll(shapes);  // draw Circle \n draw Rectangle
    return 0;
}
```

## 常见误区

- ❌ 认为虚机制是为了让函数调用更"复杂" → ✅ 虚机制是为了让设计更灵活，让接口和实现解耦
- ❌ 觉得虚函数和普通函数覆写没有本质区别 → ✅ 普通函数覆写（隐藏）在编译期绑定，虚函数在运行期动态绑定，能实现多态扩展
- ❌ 低估接口稳定性的设计价值，认为"反正也能通过其他方式实现" → ✅ 虚机制提供的是约定：基类接口稳定，派生类负责变化，客户代码无需修改
