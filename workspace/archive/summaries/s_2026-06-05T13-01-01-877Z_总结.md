## 三、薄弱环节

- **`using`声明的作用边界（唯一薄弱点）**：第2题答错反映出对"名字查找"和"对象存储"两个层面的区分不够清晰。`using` 声明作用于**作用域和名字查找**，它告诉编译器"冲突时选哪个"，但**不会改变对象的存储布局**——重复的数据成员仍然在对象中占用空间。

## 四、知识体系

```
多重继承语法（继承列表声明多个基类）
    │
    ├─ 构造/析构顺序 ──→ 继承列表声明顺序决定构造，析构逆序
    │
    ├─ 名字冲突 ──→ 作用域限定 (A::f) 或用 using 声明解决
    │                   └── ⚠️ using 只影响查找，不影响存储
    │
    ├─ 菱形结构 ──→ 共同基类重复继承 → 两份子对象 + 二义性
    │                   │
    │                   └─→ 虚基类 (virtual) 合并为一份
    │                           └── ⚠️ 代价：布局复杂 + 转换效率降低
    │
    └─ 现代设计思路 ──→ 一个带状态基类 + 多个无状态接口类
                            ├─ 接口类（纯虚函数，行为契约）
                            ├─ 工具类（静态成员，无实例变量）
                            └─ 类型标记类（仅类型名表达语义）
```
